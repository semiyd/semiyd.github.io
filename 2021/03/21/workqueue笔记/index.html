workqueue 笔记

Workqueue:
头文件：#include <linux/workqueue.h>
支持single-thread和Per-CPU thread两种形式的workqueue，对于singlethread workqueue，内核会创建单个kthread用于执行work任务，
并且该kthread不会绑定到某个CPU上；
对于Per-CPU thread workqueue，内核会在每个CPU上针对该workqueue创建一个kthread，每个kthread绑定到一个CPU上

创建work：
static struct work_struct work;
INIT_WORK(&work, work_example); //work_example是干活的函数。
static void work_example(struct work_struct *work)

创建work queue：
static struct workqueue_struct *test_wq = NULL;
test_wq = create_workqueue(“test_wq”);

启动work:
schedule_work(&work); //把work加入全局的work queue并执行。
queue_work(test_wq, &work); //把work加入自定义的work queue并执行。

停止work：
cancel_work_sync(&work)

销毁workqueue：
destroy_workqueue(test_wq);

module_exit退出的时候，之前创建的work，是没有可以马上销毁work线程的接口可以调用的。
cancel_work_sync和destroy_workqueue都会等待work线程完成后再继续执行。
注意：cancel_work()虽然在linux/workqueue.h里面extern了一下，但是没有EXPORT_SYMBOL出来，所以别的kernel module是无法直接使用的！
如果有这种需要终止正在进行的work的场景，需要用类似atomic读写的方式，靠标志位来改变work的执行流程，让其退出，见workqueue_1.c

Cmwq:
并发管理的工作队列
并发管理工作队列（cmwq）是对工作队列（wq）的重新实现，并专注于以下目标：

  1. 保持与原始工作队列API的兼容性。
  2. 使用所有等待队列共享的每CPU统一工作者线程池来提供灵活的按需并发级别，而不会浪费很多资源。
    -即：不会再因为多线程，整出一大堆线程，又是休眠状态等等，浪费线程数资源
    -旧的workqueue实现，每个workqueue都会创建一系列per cpu的thread，这会大大增加系统中的进程数量，导致系统性能的降低
  3. 自动调节工作者线程池和并发级别，以便API用户无需担心此类细节。
    -对于single thread类型的workqueue，其中的work是顺序执行，如果一个work阻塞，那么后面的都将收到影响，对于per cpu thread workqueue，情况会稍微好些，
    但是依然不够好，当一个CPU thread中已经加入了一个work时，由于queue_work只会在当前CPU上运行，
    后面再加入到该CPU thread的work依然需要等待，即使其他CPU thread是空闲的。
  4. 解决死锁的可能
    在旧版本的workqueue中，已经加入到一个CPU上的work是不能转移到另一个CPU上的，那么假设有两个work 1和work 2，work1 依赖于work2，但是work1和work2被顺序加入到了同一个CPU thread上，那么work1将永远无法等到work2执行结束，这就形成了死锁问题。
    cmwq的手段就是收编所有各个不同的源头创建的workqueue，每个要干的活被定义为一个work item。
    然后有专门的worker threads去一个一个处理这些work item。
    相当于原先公司有自己的干活的团队。现在是用项目的方式外包给外包公司的人来做了。自己没有团队了。
    CMWQ把work的生产和消费分为了两个部分，对于生产者使用workqueue来管理产生的work，消费者是使用thread pool的形式来执行work。worker pool和workqueue是一对多的关系
    生产者
    内核在启动时会创建一些系统级的workqueue，比如system_wq，除了内核自带的一些workqueue，各个模块驱动可以自行创建workqueue，这些workqueue可以通过参数来选择用什么类型的work pool来运行work。
    消费者
    CMWQ中，把用于工作的线程叫做worker，并且建立两种worker pool，也就是thread pool线程池。第一种为绑定CPU的worker pool；第二种为unbound worker pool。系统初始化时会针对每个CPU创建绑定的worker pool，每个CPU创建两个类型的worker pool，一个是普通优先级的normal worker pool，另一个是highpri worker pool。每个worker pool中包含的worker线程数量是不定的。系统也会创建unbound worker pool，也有两个类型分别是noraml和highpri。

创建work：
static struct work_struct work;
INIT_WORK(&work, work_example); //work_example是干活的函数。
static void work_example(struct work_struct *work)

创建work queue：
struct workqueue_struct alloc_workqueue(const char fmt,
unsigned int flags,
int max_active, …);
而以前旧的workqueue创建方法，当前是被cmwq兼容的。也就是用create_workqueue()创建的时候，内部实际上还是用：

#define create_workqueue(name) \
alloc_workqueue(“%s”, WQ_MEM_RECLAIM, 1, (name))
来创建的。

启动work:
schedule_work()其内部其实也是调用的queue_work：
static inline bool schedule_work(struct work_struct work)
{
return queue_work(system_wq, work);
}只不过是在system_wq上调用。
queue_work()默认是用和cpu不绑定的方式去调用的，这样在大多数情况下更为灵活，让kernel去决定。
static inline bool queue_work(struct workqueue_struct wq,
struct work_struct *work)
{
return queue_work_on(WORK_CPU_UNBOUND, wq, work);
}
通常这个接口已经够用。
如果想要绑定到特定的cpu上去调用，则可以用queue_work_on(cpu号, wq, work);
不管是用自定义的workqueue，还是用系统的workqueue，实际干活的都是kworker
PID USER PR NI VIRT RES SHR S %CPU %MEM TIME+ COMMAND
163 root 20 0 0 0 0 R 100.0 0.0 0:17.91 kworker/5:1
意思是5号cpu上的1号worker
如果使用了WQ_HIGHPRI，则可以看到是高优先级的worker来干活，NI值很低：
PID USER PR NI VIRT RES SHR S %CPU %MEM TIME+ COMMAND
234 root 0 -20 0 0 0 R 100.0 0.0 0:14.04 kworker/5:1H

停止work：
cancel_work_sync(&work)

销毁workqueue：
destroy_workqueue(test_wq)

有关alloc_workqueue的第二个入参flags：
WQ_UNBOUND说明其work的处理不需要绑定在特定的CPU上执行
WQ_FREEZABLE表示本work的执行是可以被冻住的
WQ_MEM_RECLAIM这种类型的workqueue会创建备份worker，当发生了内存回收导致worker创建失败时就使用备用的worker来进行调度
WQ_HIGHPRI说明挂入该workqueue的work是属于高优先级的work
WQ_CPU_INTENSIVE说明挂入该workqueue的work是属于特别消耗cpu的那一类，所以当系统检测到该work正在运行时，那么如果还有其他work在排队，那么会假设本worker pool已经没有可用的线程了，需要创建多余worker线程去处理其他的work。
