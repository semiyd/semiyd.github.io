Regmap 子系统

=============================regmap介绍==============================
regmap是什么？
针对i2c，spi总线，或者直接内存地址的寄存器的操作，内核做了一个抽象层，隐藏了总线号，总线地址，寄存器地址和寄存器值的位数，寄存器范围，寄存器默认值，总线速度等细节
抽象出来了一个简单的专用于操作寄存器的接口，像这样：
regmap_write(struct regmap map, int reg, int val); // 向reg写入val
regmap_read(struct regmap map, int reg, int *val); // 读取reg的数据到val中
这样可以提高代码重用性。
除此之外，regmap还能提供锁机制，cache机制（可以积攒一段时间，一次性写入，如果对时序无要求），处理irq。
包装位操作，检查寄存器范围，轮询接口包装
内核方面，代码在drivers/base/regmap/

为什么mfd会扯到regmap？
因为一个mfd内部的数个cell，访问这些cell的寄存器的方法，一般来说是相同的。例如都是同一路i2c，或者都是同一路spi，或者都是一个寄存器的内存空间的。
所以这些个cell，在访问寄存器上，有很大的共性。cell之间的区别可能只是寄存器的地址和值不一样。
所以有必要在mfd里面，利用是regmap的这个功能，这样提高了代码的重用性。

可以看到，手法一样，通过parent取出struct atmel_hlcdc私有结构体（里面有mfd设备的regmap）hlcdc，放入自己的platform驱动的私有结构体chip中
最后，真正使用regmap访问这个cell的寄存器：
static int atmel_hlcdc_pwm_set_polarity(struct pwm_chip c,
struct pwm_device pwm,
enum pwm_polarity polarity)
{
struct atmel_hlcdc_pwm chip = to_atmel_hlcdc_pwm(c);
struct atmel_hlcdc hlcdc = chip->hlcdc;
u32 cfg = 0;

if (polarity == PWM_POLARITY_NORMAL)
    cfg = ATMEL_HLCDC_PWMPOL;

return regmap_update_bits(hlcdc->regmap, ATMEL_HLCDC_CFG(6),
              ATMEL_HLCDC_PWMPOL, cfg);

}
可以看到，从自己的platform驱动的私有结构体,取出regmap。将其作为入参，使用regmap_update_bits来访问寄存器。
这个过程中，不再需要重复去ioremap，不用去关注如何访问寄存器本身，只管读写就是了。
