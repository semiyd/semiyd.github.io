i2c 笔记

i2c 笔记

I2c的驱动架构图，见Linux\Kernel\驱动\I2C\i2c驱动层次图.png

总的来说I2C的驱动分成三大部分。
一部分是向I2C子系统注册cpu的i2c控制器的驱动，也就是adapter。sysfs里面在/sys/class/i2c-adapter/下面可以看到已有的adapter。代码位于/drivers/i2c/buses
另一部分是针对具体的i2c外设的驱动，也就是client(有时在这个地方也被叫做device)。代码位于/drivers/i2c/chips
还有一个就是规定具体如何通信的协议部分，也就是algorithm。代码位于/drivers/i2c/algos

=================================================================================================
常用结构体

adapter类：
platform_device{} 定义.name(和platform_driver的.name匹配)，.id，.platform_data（定义引脚，工作频率，超时时间等）
platform_driver{} adapter的驱动结构体。
i2c_adapter{} .owner（一般为THIS_MODULE）,.algo_data指向一个自定义的结构体（结构体内可以定义通信的算法和时延等细节，可从platform_data处取得）
.class（一般为I2C_CLASS_HWMON | I2C_CLASS_SPD），.dev.parent（一般为platform_device的.dev）
.nr就是（一般为platform_device的.id），.list_head clients所连接的client的链表（已过时？）

client/device类：
i2c_board_info{} 这个是client结构体
定义了client的名字.type,i2c从设备地址.addr，还有platform_data（不知这个有什么用）
这个不属于platform_device，因为不是片上的设备。
i2c_client{} 定义client的id也就是.name，i2c从设备地址.addr，包含device结构体的定义。
.adapter指向对应的adapter
和i2c_board_info{}什么关系？关系就是，使用i2c_board_info{}调用i2c_register_board_info()，在这个过程中会创建相应的i2c_client{}
i2c_driver{} i2c_client{}的驱动结构体。
新版：
定义了client name，probe函数，remove函数，id_table用来匹配client，记录了client的id
id_table包含了i2c_device_id结构体，里面的.name就是id。这个类似于platform_driver，只是这个是client的东西，不是adapter的。
这个id_table的id只是个名字，不是从设备的地址。这个名字只要和i2c_board_info{}里面的名字相同即可。
旧版：
定义.name，.id设备地址，.attach_adapter，.detach_adapter
i2c_device_id{}

algorithm类：
i2c_msg{} 具体的消息。定义设备地址.addr，.flags标志，.len消息长度，.buf消息数据。如果是针对某个reg寄存器的读写，则一般reg被附带在i2c_msg的buf[0]了
i2c_algorithm{} 定义master_xfer()

和functionality()（返回适配器支持协议，如I2C_FUNC_I2C | I2C_FUNC_SMBUS_EMUL | I2C_FUNC_PROTOCOL_MANGLING ）

================================================================================================
常用API

adapter类：
i2c_add_adapter()，i2c_add_numbered_adapter()（被 i2c_bit_add_numbered_bus()包含）和i2c_del_adapter() 注册adapter
platform_driver_register()
platform_add_devices()

client/device类：
i2c_register_board_info() 用来注册client的信息。具体就是用来注册一个或者数个i2c_board_info{}的结构体作为这个函数的入参。
i2c_new_device() 和i2c_register_board_info()平行的一个API,目的是一样的，都是注册client。
i2c_register_board_info()是静态注册，i2c_new_device()是动态注册client。
i2c_register_board_info()必须在编译内核的时候知道i2c的总线编号和物理的连接。
感觉这种动态的注册API不会常用，因为i2c的总线编号和物理的连接总是事先知道的。
http://blog.csdn.net/tangkegagalikaiwu/article/details/8563392
I2C_BOARD_INFO() 在i2c_board_info{}里面定义的时候使用，定义type（就是个client名字）和addr（client地址）
i2c_add_driver() 注册client的驱动。
attach_adapter()旧版API。 在调用i2c_add_driver()会触发回调函数调用这个attach_adapter()
若内核中注册了i2c适配器，就顺序调用这些适配器来连接i2c设备
如果可以访问相应地址的设备，则建立i2c的设备的文件节点MKDEV
用i2c_attach_client()来注册client
注册新的字符设备，和相应的fops
detach_adapter()旧版API。

algorithm类：
master_xfer() 用于产生 I2C 访问周期需要的信号，以 i2c_msg{}为单位，将i2c_msg{}消息传给I2C设备
这个master_xfer()是如何被调用的？
用户空间open()
ioctl()
i2c_transfer()
master_xfer()
中断方式（s3c2410的）的具体的工作举例说明这个master_xfer()：
将适配器设置为主机发送状态
启动适配器的中断号，允许适配器发出中断
启动适配器的消息传输，发一个从机地址。
wait_event_timeout()等待超时或者中断函数在i2c->msg_num == 0时唤醒这里的等待队列
PS:具体中断函数的处理：
中断第一次进入在写入设备地址后（以s3c2410为例）
处理总线仲裁的异常
维护一个总线状态的状态机，不同的状态有IDLE,START,STOP,WRITE,READ
传输一个个字节。
如果传输完毕，调用wake_up()唤醒master_xfer()，完成传输
普通方式的工作举例：
static int i2c_adapter_xxx_xfer(structi2c_adapter adap, struct i2c_msg msgs, int num)
{
for (i = 0; i < num; i++)
{
产生起始位();
if (msgs[i]->flags & I2C_M_RD)/读取/
{
i2c_adapter_xxx_setaddr((msg->addr << 1) | 1); /发送从设备地址/
获得从设备的ACK();
i2c_adapter_xxx_readbytes(msgs[i]->buf,msgs[i]->len); /读取len长度的数据到buf中/
}
else/写入/
{
i2c_adapter_xxx_setaddr(msg->addr << 1); /发送从设备地址/
获得从设备的ACK();
i2c_adapter_xxx_writebytes(msgs[i]->buf, msgs[i]->len); /写入len长度的数据到buf中/
}
}
产生停止位();
}
i2c_transfer()
寻找到i2c_adapter对应的i2c_algorithm，并使用i2c_algorithm的master_xfer()函数完成传输。
这个函数，int i2c_transfer(struct i2c_adapter adap, struct i2c_msg msgs, int num)
i2c_transfer()的num，是指独立的i2c的包的个数。（一个START条件和一个结束条件）
如果说是想在一次连续的i2c传输里面传输数个字节。那么num应该=1，具体的字节数，反映在msgs[0].len
i2c_master_send()
i2c_master_send()内部会调用i2c_transfer()函数完成一条写消息。
i2c_master_recv() 

i2c_master_recv()内部会调用i2c_transfer()函数完成一条读消息。

在调用i2c_add_driver()会触发回调函数调用.probe
比较client的名字和id_table中名字，如果相等，则探测到i2c设备，
则建立i2c的设备的文件节点MKDEV。
如果需要可以建立sysfs的节点sysfs_create_bin_file（作用是在/sys/bus/i2c/devices/会生成一个文件，操作文件就等于操作硬件）

旧版：

在调用i2c_add_driver()会触发回调函数调用这个attach_adapter()
attach_adapter()里面必须调用i2c_probe()来探测设备。而i2c_probe()有三个入参
一个是adapter信息，二是地址信息，三是自定义的attach的函数
如果地址信息与芯片对应不上，无法探测到设备，当探测到目标设备后，调用自定义的attach的函数，把探测到得地址作为参数传入。
在自定义的attach函数里：
如果可以访问相应地址的设备，则建立i2c的设备的文件节点MKDEV
新建i2c_client{}结构体

注册新的字符设备，和用cdev_init()注册相应的fops
同时要实现client的file_operations各个函数。填充i2c_client{}结构体，并赋给open()的时候的struct file的private_data成员
后续实现ioctl的时候，可以把i2c_client再从private_data调出来。用copy_from_user把消息从arg的参数里面拷到内核，读取的信息再用
copy_to_user返回用户空间

读写一个字节：
参考文档：Documentation/i2c/smbus-protocol
s32 i2c_smbus_write_byte_data(const struct i2c_client *client, u8 command, u8 value)
参数：
client：传入的i2c_client
command：寄存器/命令字节
value：写入的字节的值
返回： 成功=0 不成功=负值

        这个函数的效果就是发出三个字节。一个地址字节，一个是写入的寄存器/命令字节，一个是要写入的数值
        This writes a single byte to a device, to a designated register. The
        register is specified through the Comm byte. This is the opposite of
        the Read Byte operation.
        时序：（注意[]是指对方发出的。A是指ack）
        S Addr Wr [A] Comm [A] Data [A] P

    s32 i2c_smbus_read_byte_data(const struct i2c_client *client, u8 command)
        参数：
            client：传入的i2c_client
            command：寄存器/命令字节
        返回：    成功=读到的字节    不成功=负值

        这个函数的效果就是发出地址，发出寄存器/命令值，然后再发出地址，再读入一个字节
        This reads a single byte from a device, from a designated register.
        The register is specified through the Comm byte.
        时序：（注意[]是指对方发出的。A是指ack）
        S Addr Wr [A] Comm [A] S Addr Rd [A] [Data] NA P

    s32 i2c_smbus_read_byte(const struct i2c_client *client)
        参数：
            client：传入的i2c_client
        返回：    成功=读到的字节    不成功=负值

        这个函数的效果就是发出地址，再读入一个字节
        This reads a single byte from a device, without specifying a device
        register. Some devices are so simple that this interface is enough; for
        others, it is a shorthand if you want to read the same register as in
        the previous SMBus command.
        时序：（注意[]是指对方发出的。A是指ack）
        S Addr Rd [A] [Data] NA P

    s32 i2c_smbus_write_byte(const struct i2c_client *client, u8 value)
        参数：
            client：传入的i2c_client
            value：写入的字节的值
        返回：    成功=0    不成功=负值

        这个函数的效果就是发出两个字节。一个地址字节，一个是要写入的数值
        This operation is the reverse of Receive Byte: it sends a single byte
        to a device.  See Receive Byte for more information.
        时序：（注意[]是指对方发出的。A是指ack）
        S Addr Wr [A] Data [A] P                            

    以上byte_data和byte的区别：
        以write为例：
            byte是发一个字节
            byte_data是发一个命令，紧接着发一个字节
        以read为例：
            byte是收一个字节
            byte_data是发一个命令，紧接着收一个字节

======================有关i2c操作，无法在中断上下文执行的问题============================================================

这样做直接就导致了i2c的操作，无法用于中断上下文（硬中断或软中断）

例如：
1.不能在request_irq的中断处理函数里面，操作i2c

2.不能在setup_timer注册的回调函数里面，操作i2c(软中断)

======================i2c_check_functionality的使用==========================
作用是在i2c_client驱动中，用来确认i2c控制器是否支持某种特定的i2c操作。直接看例子：
static int
lm75_probe(struct i2c_client client, const struct i2c_device_id id)
{
struct device dev = &client->dev;
struct device hwmon_dev;
struct lm75_data *data;
int status, err;
enum lm75_type kind;

if (client->dev.of_node)
    kind = (enum lm75_type)of_device_get_match_data(&client->dev);
else
    kind = id->driver_data;

if (!i2c_check_functionality(client->adapter,
        I2C_FUNC_SMBUS_BYTE_DATA | I2C_FUNC_SMBUS_WORD_DATA))//检查是不是支持字节操作和WORD操作
    return -EIO;
...

}
======================i2c_check_functionality的使用==========================

======================designware i2c控制器代码解析==========================
(ic_sda_tx_hold对应i2c spec的tHD;DAT
ic_sda_rx_hold对应scl下降沿来之前，接收方拖住sda的时间，以保证能安全度过scl下降沿的过渡时期，否则如果发送方拉高sda但下降过早，导致误判成start，导致tx arbitration lost。
