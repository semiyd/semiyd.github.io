udev简介

udev简介

Udev：
在用户空间管理设备的子系统。主要功能有：确定具体物理设备和设备节点的对应关系。支持热插拔和冷插拔。
1.物理设备和设备节点的对应关系
包括udevd守护进程和udevinfo
规则放在/etc/udev/rules.d/
udev出现之前，设备的编号如sda0,sda1只和插入的顺序有关，设备文件节点名和设备本身无对应关系。
使用方法：
1.用udevinfo来搜集设备信息，
2./etc/udev/rules.d/xxx.rules
其他的一些工具：
udevtest 检查rules文件的语法，用来调试rules文件
udevstart 刷新下udev，让新加入的设备刷出来
udevmonito 用来查看uevent
udevadm 查看/dev下某设备的一些信息，如sysfs信息
2.支持热插拔和冷插拔
udevd从kernel收到一个设备的uevent，以此决定是否要删除/dev下对应的设备文件，或者是根据配置文件/etc/udev/udev.conf，按照设定的规则，在/dev下
生成新的设备文件。
udev.conf，可以根据SYMLINK关键字，来顺便生成一个自定义名字的设备名。比如/dev/kingstonDisk。其实是指向/dev/sdc的符号链接。
这样就解决了一个问题，就是同一个设备，因为对比同类型设备插入的顺序不同，会导致/dev/下名称不固定的情况，比如一会是sda1，一会是sda2
有个SYMLINK，就可以固定为kingstonDisk。
通过MODULE_DEVICE_TABLE的宏，可以让udev在bus热插入相应的设备的时候，自动加载对应的驱动模块

http://www.ibm.com/developerworks/cn/linux/l-cn-udev/

udev工作流程：
1.所有的modules都放在/lib/modules/uname -r/下面
2.modules默认不会自动加载。可以手动加载（insmod/modprobe）。可以配置自动加载（udev）
3.热插拔时（主要指usb等设备）自动加载的情况：
设备被插入，驱动子系统层（例如usb子系统，只是子系统，不负责具体的设备功能的驱动）读了device id和vendor id，内核由此发出uevent。
所以udev能负责的都是能支持热插拔，拥有内核的某种子系统的（比如usb子系统）驱动。否则连子系统都没有的话，怎么通知内核呢？（因为驱动模块没加载）
udev收到内核的uevent（一种netlink socket message ），
根据/etc/udev/rules.d/下面的rules文件，创建/删除device节点，并从uevent取得设备信息传给modprobe（运行modprobe $MODALIAS）。
uevent例子如下：
“add@/class/input/input9/mouse2\0 // message
ACTION=add\0 // action type
DEVPATH=/class/input/input9/mouse2\0 // path in /sys
SUBSYSTEM=input\0 // subsystem (class)
SEQNUM=1064\0 // sequence number
PHYSDEVPATH=/devices/pci0000:00/0000:00:1d.1/usb2/2­2/2­2:1.0\0 // device path in /sys
PHYSDEVBUS=usb\0 // bus
PHYSDEVDRIVER=usbhid\0 // driver
MAJOR=13\0 // major number
MINOR=34\0”, // minor number
$MODALIAS举例：MODALIAS=usb:v046DpC03Ed2000dc00dsc00dp00ic03isc01ip02
modprobe会去查看/lib/modules/uname -r/modules.alias，来决定要不要加载某个module。
modprobe加载module后，去调用module的init_module宏。
在module加载完毕后，内核会在/sys/module下面建立相应的信息。
4.boot时，内核会枚举一下所有已有的硬件（主要指usb,pci设备），以方便启动时就加载已插入的硬件
5.boot时，内核会扫描dts.对于里面的compatible字段，如果驱动已经built-in，则自动加载。如果没有built-in，则在启动后，rootfs挂载后，
在/lib/modules下面符合compatible字段的模块会被自动加载（这也就是c_can的情况）。
6.如果要改动/lib/modules/uname -r/下面的模块。需要使用depmod -a更新/lib/modules/uname -r/modules.xxx文件
7.如果要boot的时候，强制load某个模块。可以把模块名加在/etc/modules这个文件里面。

涉及的文件：
/etc/udev/udev.conf 主要用来配置给syslog输出什么log的。
/lib/udev/rules.d/.rules udev标准的默认的rules文件
/etc/udev/rules.d/.rules 用户自定义的rules文件
/lib/udev/* 从rules里面去调用的一些可执行文件helper programs，如cdrom_id

udev实战之固定映射设备号：

现在有这么个需求。
板卡有usb host0和usb host1
无论是host0还是host1，只会被插入fdti公司的usb转串口的芯片模块。并且每个芯片插入的时候，会产生两个ttyUSB设备，也就是每路usb有两个interface。
现在要求，只要是插入host0的，一律起名为host0-xxx。只要插入host1的，一律起名为host1-xxx。
因为linux针对ttyUSB这种热插拔设备的命名顺序，是针对插拔的顺序的。谁先插入，谁就是ttyUSB0，后来的人只能叫ttyUSB1,2,3…。
解决的办法就是利用udev，建立新的rules，从而建立设备的链接。
vi /etc/udev/rules.d/99-ttyUSB.rules

SUBSYSTEM==”tty”, ATTRS{idVendor}==”0403”, ATTRS{idProduct}==”6010”, SYMLINK+=”INNCOM-PORT0”

FDTI的芯片，其idVendor是0403, idProduct是6010。这是可以通过lsusb命令得到的：
host0的lsusb结果：
Bus 001 Device 003: ID 0403:6010
host1的lsusb结果：
Bus 002 Device 003: ID 0403:6010
那么这里关键，不同的host插进来的设备，他的busnum是不一样的。
/sys/class/usb_device/usbdevx.x/device下面，可以看到busnum的属性。cat busnum就能看到当前设备所在的busnum。
host0插入的，busnum是1.host1插入的，busnum是2.
所以除了加入idVendor和idProduct的判断外，还要加入busnum的判断。

SUBSYSTEM==”tty”, ATTRS{idVendor}==”0403”, ATTRS{idProduct}==”6010”, ATTRS{busnum}==”1”, SYMLINK+=”INNCOM-PORT0”

到这一步，还是有问题。因为一个fdti芯片，有两个interface。也就是一个芯片会生成两个ttyUSB设备。
如果仅仅是上面的rules。结果是ttyUSB0没有被建立链接。建立的链接是ttyUSB1的。这样不行。

为了区分同一个芯片的不同的interface，
进一步在/sys下挖掘，发现：
类似/sys/class/usb_device/usbdev1.2/device/1-1:1.0/bInterfaceNumber 这个可以看出是哪个interface，也就是哪个ttyUSB

但是问题来了，bInterfaceNumber，不属于tty这个SUBSYSTEM，属于usb这SUBSYSTEM。

所以最终的代码是这样的：
SUBSYSTEMS==”usb”, ENV{.LOCAL_ifNum}=”$attr{bInterfaceNumber}”
SUBSYSTEM==”tty”, ATTRS{idVendor}==”0403”, ATTRS{idProduct}==”6010”, ATTRS{busnum}==”1”, SYMLINK+=”INNCOM-PORT0-%E{.LOCAL_ifNum}”
SUBSYSTEM==”tty”, ATTRS{idVendor}==”0403”, ATTRS{idProduct}==”6010”, ATTRS{busnum}==”2”, SYMLINK+=”INNCOM-PORT1-%E{.LOCAL_ifNum}”

这里，udev会遍历所有搜到的bInterfaceNumber（假设bInterfaceNumber不止一个），从而针对一个fdti芯片，也能生成两个链接文件。这也是udev比较聪明的地方。
如下：
[root@M28x ~]# ls -l /dev/IN*
lrwxrwxrwx 1 root root 7 Jan 1 00:00 /dev/INNCOM-PORT1-00 -> ttyUSB0
lrwxrwxrwx 1 root root 7 Jan 1 00:00 /dev/INNCOM-PORT1-01 -> ttyUSB1

mdev简介

mdev是udev的简化版
集成于busybox中，所以常见于嵌入式系统。
使用方法：
busybox里面勾上关于mdev的选项：
[] mdev
[] Support /etc/mdev.conf
[] Support subdirs/symlinks
[] Support regular expressions substitutions when renaming device
[] Support command execution at device addition/removal
[] Support loading of firmwares
开机rcS脚本运行：
echo /sbin/mdev > /proc/sys/kernel/hotplug
mdev -s
然后在/etc/mdev.conf里面配置：
mmcblk[0-9] 0:0 0660 @(mount -t vfat /dev/$MDEV /mnt)
mmcblk[0-9] 0:0 0660 *(umount /mnt)
这样的话，当mmc卡插入，/dev/mmcblk0设备会生成，然后mdev就可以自动把mmcblk0挂载到/mnt目录下。
