SPI 笔记

下面分为SPI主控部分驱动，SPI从设备部分驱动两部分来详解SPI的驱动细节.最后看下spi用户空间的接口SPIDEV的细节。

1.SPI主控部分驱动（以drivers/spi/spi-omap2-mcspi.c为例）

主控部分一般都是用platform的架构，先后注册platform_device和platform_driver
然后在platform的probe里面，最终调用
int spi_register_master(struct spi_master *master)
来注册spi的主控制器的驱动。
具体步骤一般如下：

a.通过struct spi_master spi_alloc_master(struct device dev, unsigned size)分配产生spi_master{}结构体
这步的作用是生成spi_master{}结构体
参数*dev就是platform_device->dev
参数size是驱动私有结构体的大小（以am335x为例就是struct omap2_mcspi的大小）
这个函数如果看起内部的代码的话，做了这几件事情：
分配空间，空间大小是spi_master的大小+驱动私有结构体的大小
生成spi主控的device{}结构体，并将其指针放入spi_master{}
设置spi主控的device{}结构体的->p->driver_data也就是device结构体的私有数据指针为刚刚分配到的地址。
返回spi_master{}指针
这个函数，分配的空间，刚刚提到是由两部分组成的。也就是spi_master的大小+驱动私有结构体的大小
第一部分由直接函数返回值返回的spi_master指针。
第二部分是根据size分配驱动私有结构体所需的空间，并且将这个分配的私有结构体指针，指向dev->p->driver_data也就是device结构体的私有数据指针。
注意此处的dev->p->driver_data所指的dev，是指spi_alloc_master新生成的spi主控的device，并非platform_device->dev。
也就是这里有两个device{}结构体。一个是spi_alloc_master新生成的spi主控的device，一个是早已存在的platform_device->dev
这两个device{}的关系是，platform_device->dev是spi_alloc_master新生成的spi主控的device的父类。代码上看是spi_master->dev.parent = get_device(dev);。
而此时父类platform_device->dev的->p->driver_data，其实尚未初始化。
所以经过spi_alloc_master，我们可以分配得到连个指针。一个是spi_master{}的，另外一个是spi_master->dev->p->driver_data也就是spi device结构体的私有数据指针。

b.继续填充spi_master{}结构体（结构体定义于include\linux\spi\spi.h）
此处以具体的代码为例解释
master->mode_bits = SPI_CPOL | SPI_CPHA | SPI_CS_HIGH; SPI模式的配置到mode_bits
if (pdev->id != -1) 如果platform设备不是只有一个的话
master->bus_num = pdev->id; 需要把设备号赋值到bus_num里面
master->setup = omap2_mcspi_setup; 注册setup()成员函数。内容包括申请DMA，使能SPI的控制器的时钟源，配置SPI传输的各个参数
master->transfer = omap2_mcspi_transfer; 注册transfer()成员函数。类似于i2c的master_xfer()，是传数据的函数。
master->cleanup = omap2_mcspi_cleanup; 注册cleanup()成员函数。内容是释放DMA等清理工作。
master->num_chipselect = pdata->num_cs; 把platform_device->dev.platform_data里面的CS线的数目，赋值到num_chipselect成员。
此处可能会觉得奇怪，为什么CS线的数目都要规定？（比如说是2）
一般的单片机系统，其实SPI的CS是随便搞个GPIO拉一拉的。
但此处am335x平台为例，其实SPI的CS也是专用的脚，并非gpio，mux配置的时候，配置的是专用的功能SPI_CSn。
所以此处就有了，SPI究竟支持几个CS的规定。
那么回过来，platform_device->dev.platform_data->num_cs是从哪里来的呢？
在注册SPI主控的platform_device的时候，可以看到pdata->num_cs = mcspi_attrib->num_chipselect;
那么在arch\arm\mach-omap2\omap_hwmod_33xx_data.c，就定义了.num_chipselect = 2。
对于am335x来说，SPI0主控有CS0和CS1。SPI1主控也有CS0和CS1。

c.通过dev_set_drvdata()，把platform_device->dev->p->driver_data，设置成指向spi_master{}结构体

d.通过static inline void spi_master_get_devdata(struct spi_master master)，获取spi_master{}的dev的->p->driver_data，也就是具体到am335x spi驱动是omap2_mcspi{}
当然此时的omap2_mcspi{}还是空的。（刚说过由spi_alloc_master()分配空间）
后面还要对这个结构体的成员做具体的初始化。

e.初始化omap2_mcspi{}私有结构体。
此结构体定义如下。可以看到里面基本上是一个集合，包含了工作队列，锁，消息队列，dma，寄存器地址之类的驱动常用的数据。
struct omap2_mcspi {
struct work_struct work;
/ lock protects queue and registers /
spinlock_t lock;
struct list_head msg_queue;
struct spi_master master; 赋值为spi_master{}指针
/ Virtual base address of the controller /
void __iomem base;
unsigned long phys;
/ SPI1 has 4 channels, while SPI2 has 2 /
struct omap2_mcspi_dma dma_channels;
struct device dev;
};
具体初始化如下：
master成员赋值为spi_master{}指针
phys成员赋值为spi寄存器物理起始地址
base成员赋值为spi寄存器虚拟起始地址
platform_get_resource
request_mem_region
ioremap
dev成员赋值成platform_device->dev
work成员用INIT_WORK()初始化一个工作队列
lock成员用spin_lock_init()初始化自旋锁
msg_queue成员用INIT_LIST_HEAD()初始化链表头
dma_channels成员用platform_get_resource_byname()获取dma的channel号。具体每个CS都有两个DMA(rx+tx)，所以两个CS就是有4个DMA.

f.pm_runtime_enable使能

g.int spi_register_master(struct spi_master *master)
向SPI子系统注册spi_master{}

2.SPI从设备部分驱动

spi_board_info{}注册spi_device{}
spi_register_driver()注册spi_driver{}
然后spi_board_info.modalias需要和spi_driver{}的driver.name匹配。触发spi_driver{}的probe()

具体的spi_board_info{}的成员解释：
struct spi_board_info {
char modalias[SPI_NAME_SIZE]; //spi_board_info.modalias需要和spi_driver{}的driver.name匹配
const void platform_data; //私有的配置用的数据结构，将来可以带入到spi_driver{}的probe()里面去使用的。
void controller_data; //这个是给spi_master_driver{}用的，用来配置针对特定的spl slave设备的信息。比如DMA，字长之类的
int irq; //
u32 max_speed_hz; //SPI总线速度。
u16 bus_num; //意指连接第几个spi的主控（如果mcu有数个spi主控）。比如连接spi1。或者连接spi2.
u16 chip_select; //意指是spi的cs几
u8 mode; //配置CPHA和CPOL这种参数
};

读写msg的具体方法：
以具体代码为例举例：

struct spi_device spi = to_spi_device(chip->gpio_chip.dev);
struct spi_message message;
struct spi_transfer msg_buf;
int i, ret = 0;
u8 spi_transfer_rxbuf[2] = {0};

mutex_lock(&chip->lock); //读写之前上锁

msg_buf = kzalloc(chip->registers * sizeof(struct spi_transfer), //分配spi_transfer结构体
GFP_KERNEL);
if (!msg_buf)
return -ENOMEM;

spi_message_init(&message); //初始化spi_message结构体

/*

  • Since the registers are chained, every byte sent will make
  • the previous byte shift to the next register in the
  • chain. Thus, the first byte send will end up in the last
  • register at the end of the transfer. So, to have a logical
  • numbering, send the bytes in reverse order so that the last
  • byte of the buffer will end up in the last register.
    */
    for (i = chip->registers - 1; i >= 0; i–) {
    msg_buf[i].tx_buf = chip->buffer + i; //设置spi_transfer的tx_buf指针位置
    msg_buf[i].rx_buf = spi_transfer_rxbuf + i; //设置spi_transfer的rx_buf指针位置
    msg_buf[i].len = sizeof(u8);
    spi_message_add_tail(msg_buf + i, &message); //把设置好的spi_transfer加入队列
    }

ret = spi_sync(spi, &message); //开始收发spi信息。在spi_sync()以后，rx_buf会被更新。

kfree(msg_buf);

mutex_unlock(&chip->lock);

return spi_transfer_rxbuf[1];
