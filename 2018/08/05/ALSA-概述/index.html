ALSA 概述

ALSA是Advanced Linux Sound Architecture 的缩写

主页：
http://www.alsa-project.org/main/index.php/Main_Page

概述：
linux的音频系统
从硬件架构上看，分为下面几个部分：
1.最底层为audio DAC/ADC，复杂一些的如声卡芯片，集成了DAC/ADC和DSP。这类芯片总称codec。
2.codec和cpu之间的音频数据，最常见的由i2s连接，也有用usb连接的。还有些控制命令，通常由另外一路i2c或者spi来连接。有时候codec也会由cpu来提供主时钟信号，或者自接一个晶振。
3.cpu这端，通常需要一个i2s的控制器。具体到am335x，是mcasp硬件控制器。
从软件架构上看，从最顶层往下，分为下面几个部分：（详见同目录下的”软件架构图.png“）
1.ALSA应用程序，例如aplay或者其他自己写的基于ALSA API的应用程序。另外还包括顶层的多媒体框架比如GStreamer和PulseAudio。
2.ALSA库
3.ALSA内核子系统
4.ALSA SOC子系统
5.codec驱动，machine驱动（连接codec和platform的中间层驱动），platform驱动（即i2s控制器驱动）
ALSA是Advanced Linux Sound Architecture 的缩写

主页：
http://www.alsa-project.org/main/index.php/Main_Page

ALSA有四部分组成：

1.最底层的基于ALSA的设备驱动
包括Codec driver，比如TLV320AIC3254的驱动，如sound/soc/codecs/tlv320aic3x.c
包括Platform driver，比如ti平台的mcasp也就是i2s的驱动，如sound/soc/davinci/davinci-mcasp.c
还包括Machine driver，作用是将codec和i2s驱动连接，并且提供一些平台相关的操作比如开始播放时打开功放，如sound/soc/davinci/davinci-evm.c，或者更通用的sound\soc\generic\Simple-card.c
2.内核的ALSA子系统
为了更好的支持嵌入式系统，现在有了一个叫ASoC层的东西。
ASoC目的是
让codec驱动摆脱对特定平台代码的依赖，在ASoC之前，每个codec在每个平台都需要有分别对应的驱动。具体的说，就是定义了Machine driver这种类似胶水层的驱动层。
对Dynamic Audio Power Management (DAPM)的支持，就是说让codec的电源消耗更优化更少
dapm,就是Dynamic Audio Power Management。即动态音频电源管理，是针对移动设备的。
dapm和内核的电源管理子系统是相互独立的。
dapm相对于应用程序，是透明的，应用无需知道任何dapm细节。dapm会根据音频的播放/采集的活动等，决定电源开关。
任何影响到电源消耗的音频组件被称为widgets
消除噗噗声（优化codec内部开关顺序）
等
3.用户空间的ALSA库 alsa-lib
4.用户空间的ALSA应用程序

========================================================/dev接口========================================================
/dev/snd/controlC0 Control devices (i.e. mixer, etc)
/dev/snd/pcmC0D0c PCM Card 0 Device 0 Capture device
/dev/snd/pcmC0D0p PCM Card 0 Device 0 Playback device

========================================================/proc接口========================================================
/proc/asound下面有这些内容：
cards List of registered cards
version Version and date the driver was built on
devices List of registered ALSA devices
pcm The list of allocated PCM streams
cardX/ (X = 0-7) The card specific directory
cardX/pcm0p The directory of the given PCM playback stream
cardX/pcm0c The directory of the given PCM capture stream

========================================================用户空间API========================================================
snd_pcm_open Opens a PCM stream
snd_pcm_close Closes a previously opened PCM stream
snd_pcm_hw_params_any Fill params with a full configuration space for a PCM
snd_pcm_hw_params_test_<> Test the availability of important parameters like number of channels, sample rate etc. For e.g. snd_pcm_hw_params_test_format, snd_pcm_hw_params_test_rate
snd_pcm_hw_params_set_<> Set the different configuration parameters. For e.g. snd_pcm_hw_params_set_format, snd_pcm_hw_params_set_rate etc.
snd_pcm_hw_params Install one PCM hardware configuration chosen from a configuration space
snd_pcm_writei Write interleaved frames to a PCM
snd_pcm_readi Read interleaved frames from a PCM
snd_pcm_prepare Prepare PCM for use
snd_pcm_drop Stop a PCM dropping pending frames
snd_pcm_drain Stop a PCM preserving pending frames

========================================================用户空间程序========================================================
aplay -l list the available sound cards and PCMs for playback
arecord -l list the available sound cards and PCMs for capture
aplay -Dplughw:0,0 In most cases -Dplughw:0,0is the device we want to use for audio 播放wav文件
arecord -Dplughw:0,0 -t wav 录音wav文件
arecord -Dplughw:0,0 | aplay -Dplughw:0,0 全双工，边放边录

arecord -Dhw:0,0 -fdat -t wav record_48K_stereo_16bit.wav record 48KHz, stereo 16bit audio
arecord -Dhw:0,0 -fS24_LE -c2 -r96000 -t wav record_96K_stereo_24bit.wav record record 96KHz, stereo 24bit audio
alsactl -f board.aconf store 保存mixer setting
alsactl -f board.aconf restore 调用mixer setting（通常是开机时）

amixer -c AM335xEVM sset PCM 90 # Master Playback volume

For audio capture trough stereo microphones:
amixer sset ‘Right PGA Mixer Line1R’ on
amixer sset ‘Right PGA Mixer Line1L’ on
amixer sset ‘Left PGA Mixer Line1R’ on
amixer sset ‘Left PGA Mixer Line1L’ on

$ amixer cset name=’PGA Capture Volume’ x%,x%

========================================================用户空间程序========================================================
In case of XRUN (under or overrun)
The underrun can happen when an application does not feed new samples in time to alsa-lib(due CPU usage). The overrun can happen when an application does not take new
captured samples in time from alsa-lib.
There could be several reasons for XRUN to happen but it is usually points to system latency issues connected to CPU utilization or latency caused by the storage device.
Things to try:
1.increase the buffer size (ALSA buffer and period size)
2.try to cache the file to be played in memory
3.try to use application which use threads for interacting with ALSA and with the filesystem

=========================================================Platform driver，Codec driver，Machine driver总结=============================================================
例程：（5.2.5 kernel）
sound\soc\pxa\pxa2xx-ac97.c platform驱动，使用snd_soc_register_component注册
sound\soc\codecs\wm9713.c codec驱动，使用snd_soc_register_component注册
sound\soc\pxa\mioa701_wm9713.c machine驱动，使用devm_snd_soc_register_card注册

Platform driver
int snd_soc_register_component(struct device dev,
const struct snd_soc_component_driver component_driver,
struct snd_soc_dai_driver *dai_drv,
int num_dai)
注册component。
所谓的component，是codec和platform的共用的基础数据结构。所谓“共用”，就是component里面的有些成员，适用于给Platform驱动，另外一些成员，适用于给Codec驱动。
所以codec和platform都可以调用snd_soc_register_component来注册。
snd_soc_register_component需要两个入参：
snd_soc_component_driver和snd_soc_dai_driver。
snd_soc_component_driver 在Platform驱动中，需要定义pcm相关的操作。
snd_soc_dai_driver 所谓dai，就是digital audio interface，也就是数字音频界面。用于描述数字音频接口的属性，例如采样率，格式。
下面详细展开snd_soc_component_driver和snd_soc_dai_driver。
snd_soc_component_driver在Platform驱动中，需要定义：
.ops =, 需要定义一系列的snd_pcm_ops，用来配置soc侧的音频总线控制器。
.pcm_new = , 一般是分配DMA空间
.pcm_free = , 一般是释放DMA空间
snd_soc_dai_driver需要定义：
.playback = {
.stream_name = ,
.channels_min = ,
.channels_max = ,
.rates = ,
.formats = },
.capture = {
同playback},
.ops = , 需定义snd_soc_dai_ops的实现。

Codec driver
这个有两种注册方式。早期的内核使用snd_soc_register_codec来注册。（此API现已废弃）
较新的内核则和Platform driver一样，需要填充snd_soc_component_driver，然后使用snd_soc_register_component注册。
下面一一来看。
snd_soc_register_codec因为已不用了，说的简单点：
入参需要snd_soc_codec_driver和snd_soc_dai_driver两个。snd_soc_dai_driver在上面Platform driver的时候已经讲过了。
snd_soc_codec_driver需要实现codec芯片本身一系列的读写操作函数，例如使用i2c对codec的读写接口。
snd_soc_register_component在Codec驱动中，需要定义：（以wm9713.c为例）
static const struct snd_soc_component_driver soc_component_dev_wm9713 = {
.probe = wm9713_soc_probe,
.remove = wm9713_soc_remove,
.suspend = wm9713_soc_suspend,
.resume = wm9713_soc_resume,
.set_bias_level = wm9713_set_bias_level,
.controls = wm9713_snd_ac97_controls, 定义snd_kcontrol_new类型的控件
.num_controls = ARRAY_SIZE(wm9713_snd_ac97_controls),
.dapm_widgets = wm9713_dapm_widgets, 定义snd_soc_dapm_widget
.num_dapm_widgets = ARRAY_SIZE(wm9713_dapm_widgets),
.dapm_routes = wm9713_audio_map, 定义snd_soc_dapm_route
.num_dapm_routes = ARRAY_SIZE(wm9713_audio_map),
.idle_bias_on = 1,
.use_pmdown_time = 1,
.endianness = 1,
.non_legacy_dai_naming = 1,
}
可以看到，虽然同为snd_soc_register_component，但需要定义的内容和Platform driver有很大不同，体现在：
需要定义一系列的控件和电源管理内容，包括：
snd_kcontrol_new 参见笔记kcontrol.docx
snd_soc_dapm_widget 参见dapm笔记.txt
snd_soc_dapm_route 参见dapm笔记.txt
那之前snd_soc_codec_driver注册的那一系列i2c的读写函数，在新API snd_soc_register_component中，跑哪里去了呢？
答案在snd_soc_register_component的probe中。
通过snd_soc_component_init_regmap，向ALSA注册regmap信息，提供了寄存器读写的方法函数。
在通过regmap注册好以后，就可以使用alsa的API来读写寄存器了：(定义于sound/soc/soc-io.c)
snd_soc_component_update_bits
snd_soc_component_write
snd_soc_component_read
以上的API,在有regmap的情况下，会优先使用regmap接口。
如果没有用snd_soc_component_init_regmap，也行，只要注册了snd_soc_component_driver.read和snd_soc_component_driver.write，以上读写API也是可以工作的。

Machine driver
int devm_snd_soc_register_card(struct device dev, struct snd_soc_card card)
注册声卡设备，这个函数的输入是snd_soc_card结构体。
snd_soc_card通常需要定义如下的数个重要成员：
.name = , 对应将来生成的声卡的名字
.dai_link = , 类型是snd_soc_dai_link，作用是链接Codec和Platform驱动的snd_soc_dai_driver。
.dapm_widgets = ,
.dapm_routes = ,
dai_link通常需要定义如下的数个重要成员：
.name = ,
.stream_name = ,
.cpu_dai_name = , 对应platform驱动的snd_soc_dai_driver.name
.codec_dai_name = , 对应codec驱动的snd_soc_dai_driver.name
.codec_name = , 对应codec驱动的platform_driver.name
.init = ,
.platform_name = ,
.ops = , 对应snd_soc_ops，可以为空，因为ALSA有默认的实现。
除了主动调用devm_snd_soc_register_card，
也可以注册一个”soc-audio”的platform_device，在soc-core.c里面，有对应的platform_driver，从而调用soc_probe，最终去调用snd_soc_register_card
