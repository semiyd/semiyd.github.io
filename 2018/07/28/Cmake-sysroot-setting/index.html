Cmake sysroot setting

例子：
cd “${BUILD_DIR}” && cmake -D CMAKE_TOOLCHAIN_FILE=../arm_linux_toolchain.cmake ..

例子2：
cmake .
make

交叉编译的例子：
cmake -D CMAKE_TOOLCHAIN_FILE=arm_linux_toolchain.cmake .
设置交叉编译器的PATH
make
arm_linux_toolchain.cmake已放在当前目录中

有关使用arm-poky-linux-gnueabi-gcc-5.3.0编译的时候，遇到的设置sysroot的方法的解决方案：
参见CMake Cross-Compilation Based on Yocto SDK.pdf和cpu.tar.bz2（参见其中的arm_linux_toolchain.cmake和CMakeLists.txt）
总的来说，针对cmake，想要设置sysroot这个参数，方法如下：
想要设置这个参数，必须设置在arm_linux_toolchain.cmake里面，也就是CMAKE_TOOLCHAIN_FILE指向的文件。设置在CMakeLists.txt是无用的。
针对sysroot,cflag,ldflag，分别是以下几个变量进行设置：
CMAKE_SYSROOT “${CMAKE_FIND_ROOT_PATH}/cortexa7hf-neon-poky-linux-gnueabi”)
set(CMAKE_C_FLAGS “-Wall -Wextra -s -mfloat-abi=hard –sysroot=/home/share/cross-compiler/arm-poky-linux-gnueabi-gcc-5.3.0-MYD-Y6ULX/sysroots/cortexa7hf-neon-poky-linux-gnueabi” CACHE STRING “” FORCE)
set(CMAKE_CXX_FLAGS “-std=c++11 -Wall -Wextra -s -mfloat-abi=hard –sysroot=/home/share/cross-compiler/arm-poky-linux-gnueabi-gcc-5.3.0-MYD-Y6ULX/sysroots/cortexa7hf-neon-poky-linux-gnueabi” CACHE STRING “” FORCE)
set(CMAKE_CXX_FLAGS_DEBUG “-g -mfloat-abi=hard –sysroot=/home/share/cross-compiler/arm-poky-linux-gnueabi-gcc-5.3.0-MYD-Y6ULX/sysroots/cortexa7hf-neon-poky-linux-gnueabi” CACHE STRING “” FORCE)
set(CMAKE_CXX_FLAGS_RELEASE “-O2 -mapcs-frame -mfloat-abi=hard –sysroot=/home/share/cross-compiler/arm-poky-linux-gnueabi-gcc-5.3.0-MYD-Y6ULX/sysroots/cortexa7hf-neon-poky-linux-gnueabi” CACHE STRING “” FORCE)
set(CMAKE_EXE_LINKER_FLAGS “${CMAKE_EXE_LINKER_FLAGS} -mfloat-abi=hard –sysroot=/home/share/cross-compiler/arm-poky-linux-gnueabi-gcc-5.3.0-MYD-Y6ULX/sysroots/cortexa7hf-neon-poky-linux-gnueabi” CACHE INTERNAL “” FORCE)
set(CMAKE_SHARED_LINKER_FLAGS “${CMAKE_SHARED_LINKER_FLAGS} -mfloat-abi=hard –sysroot=/home/share/cross-compiler/arm-poky-linux-gnueabi-gcc-5.3.0-MYD-Y6ULX/sysroots/cortexa7hf-neon-poky-linux-gnueabi” CACHE INTERNAL “” FORCE)

有关链接动态库的语法：
1.链接的是同一个cmake工程内部的so：不要用link_directories和link_libraries，直接再add_library或add_executable后面加
target_link_libraries，那个被链接的so，需要在add_library时候声明下SHARED，例如add_library($(module) SHARED $(libsources))
而module定义为例如set (module ABC)
在target_link_libraries内部用这个module名就可以，假设当前是一个$(APP)需要链接ABC这个so：
target_link_libraries ( $(APP) ABCD)
注意：target_link_libraries的目标，不能包含当前cmake工程内部即将生成的.so，否则会导致cmake预处理失败。也就是不能自己link自己要产生的so。
2.链接的是外部工程目录的so：用link_directories和link_libraries，然后仍然要用target_link_libraries，但里面要写库文件的全名：
target_link_libraries ( $(APP) libABCD.so)
3.上述两种情况，均需要遵循规则：先设定include_directories,link_directories和link_libraries，再设定add_library或add_executable，
最后设定target_link_libraries的顺序
参考：
《CMake使用总结(一).pdf》
《cmake 的link_libraries和target_link_libraries.pdf》

有关propagate传递的规则：
假设cmake工程
/CMakeLists.txt
/a/CMakeLists.txt
有子目录a，根目录的CMakeLists.txt用add_subdirectory(a)包含了子目录的CMakeLists.txt
则默认/CMakeLists.txt里面的所有变量，包括include_diretories，link_directories都会传递到目录的CMakeLists.txt里面，也就是逐级继承。
所以如果子目录不想用根目录的include_diretories怎么办？
这种情况下，在/CMakeLists.txt里面就不要用include_diretories，而是要换成target_include_diretories，并用PRIVATE属性。
然后/a/CMakeLists.txt也用自己的target_include_diretories就可以了。
注意：一旦/CMakeLists.txt定义了include_diretories，/a/CMakeLists.txt即使用了target_include_diretories也无济于事，还是会继承/CMakeLists.txt的定义
有关PRIVATE,PUBLIC,INTERFACE的详细解释，见《cmake：target___ 中的 PUBLIC，PRIVATE，INTERFACE - 知乎》
另外在传递例如链接的路径的时候，/CMakeLists.txt注意不要用相对寻址，像../../usr/lib，这样传递到子目录，子目录仍然用../../usr/lib，就不对了，还是
要用绝对地址的寻址方式，在cmake从外部开始执行的时候，通过-D参数传入当前的pwd对应的变量.
