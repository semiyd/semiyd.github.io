shell脚本执行顺序的注意点

Posted on 2018-07-25 | In Linux , Application , Shell

假设有这么个脚本，作为某应用程序的启动脚本。

#!/bin/sh

#Start up Application
ps -ef| grep “eN-Display” | grep -v “grep”
isrunning=$?
if [ ! “$isrunning” -eq 0 ]; then
./eN-Display
fi

那么现在有另外一端脚本。需要执行上面的启动脚本，并等待eN-Display真正运行起来。

/home/DSP/dspd &
until [ $(pidof eN-Display | wc -w) -gt 0 ] #wait until eN-Display is up before re-check if it’s alive again with the while loop
do
sleep 1
done

这段代码实际执行是有问题的。
原因是：
/home/DSP/dspd &这句话，实际上是派生了一个新的shell脚本的进程，并立即返回。这就打破了由上往下的线性的执行关系。
所以根据具体cpu执行速度的情况，有可能还没等到dspd执行完毕，下面一句话
until [ $(pidof eN-Display | wc -w) -gt 0 ]
就先执行了。
那么比较极端的情况，就是dsps根本还没来得及开始执行，
until [ $(pidof eN-Display | wc -w) -gt 0 ]就跑了
那么until [ $(pidof eN-Display | wc -w) -gt 0 ]这句话，本质上，是要跑一个叫pidof的程序的。
这个pidof程序，从开始运行到退出，也是需要时间的。
正当pidof运行的时候，这时候dspd &脚本跑到了ps -ef| grep “eN-Display” | grep -v “grep”
会产生什么结果呢？
ps -ef| grep “eN-Display” | grep -v “grep”的本意是想看看是否eN-Display已经在跑。在跑就不要启动了。
但此时，ps -ef| grep “eN-Display” | grep -v “grep”这句话因为刚刚的pidof尚未退出，会反过来抓到这个结果：

19542 root 0:00 pidof eN-Display

这句话其实就是ps的打印信息过滤后的结果
ps的结果例如：
28498 root 0:00 {guard.sh} /bin/sh ./guard.sh
28568 root 0:00 [kworker/0:5]
29881 root 0:03 ./eN-Display -qws
29882 root 0:00 {log_check} /bin/sh ../log_check dsp

因为pidof这行里面带上了eN-Display的关键词，被误抓了。
结果，dspd误以为，eN-Display已经启动，马上决定不再启动eN-Display
最终会导致，
/home/DSP/dspd &
until [ $(pidof eN-Display | wc -w) -gt 0 ] #wait until eN-Display is up before re-check if it’s alive again with the while loop
do
sleep 1
done
有时候会并没有去启动eN-Display

解决办法是：
Restart_Display(){
/home/DSP/dspd
until [ $(pidof eN-Display | wc -w) -gt 0 ] #wait until eN-Display is up before re-check if it’s alive again with the while loop
do
sleep 1
done
}
去掉/home/DSP/dspd后面的&符号。
这样，保证dspd运行结束以后，再去检查pidof，这样就没问题了
