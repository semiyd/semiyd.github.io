基本步骤：

​ 1.用eclipse新建工程，import已有的C++代码，填入正确的include路径和link路径（注意加入-lgtest和-lgtest_main和-lpthread）。
2.在main.cpp中，加入下面的代码：

#include <gtest/gtest.h>

​ int main(int argc, char* argv) {
::testing::InitGoogleTest(&argc, argv);
return RUN_ALL_TESTS();
return 0;
}
3.新建一个cpp文件，单独用来编写各个TEST()测试用例，同一个文件可以包含不同测试。也可以建多个cpp文件包含不同的TEST()，到时候RUN_ALL_TESTS()会一并运行的。
4.编译并执行。
基本用法：
API大致分两类，ASSERT和EXPECT
ASSERT_() 属于断言类，如果是返回正确或者返回nonfatal failure则继续，否则就停止程序，打印出错位置，出错信息可定制。
如果说遇到出错，再往下执行毫无意义，则应该用ASSERT_*。

​ EXPECT_() 返回正确或者返回nonfatal failure，不影响程序继续执行。具体的种类和ASSERT_其实是一样的。以下的例子，凡是有ASSERT_，一样会有对应的EXPECT_变体。

​ TEST(TestSuite, TestCase) 用于编写测试用例。
TestSuite是一个大的类别的名称，例如”字符串测试“
TestCase是里面的测试用例的小分类，例如“字符串设置”，“字符串删除”这种。

​ 布尔值检查： ASSERT_TRUE/ASSERT_FALSE EXPECT_TRUE/EXPECT_FALSE
举例：
EXPECT_FALSE(IsPrime(4));
数值型数据检查： ASSERT_EQ/ASSERT_NE 注意当参数是指针的时候，仅比较指针的值，不比较指针内容。但可用于比较string objects。
ASSERT_LT/ASSERT_LE/ASSERT_GT/ASSERT_GE
举例：
EXPECT_EQ(3, Add(1, 2))
浮点型检查：
ASSERT_FLOAT_EQ/ASSERT_DOUBLE_EQ/ASSERT_NEAR（对相近的两个数比较）
字符串检查 ： ASSERT_STREQ/ASSERT_STRNE/ASSERT_STRCASEEQ/ASSERT_STRCASENE 比较字符串（c语言）。后面两组是忽略大小写。
有关指针，还有别的接口，用来比较子串，前缀，后缀，正则表达式。
举例：
// Can we set the MyString to NULL?
s.Set(NULL);
EXPECT_STREQ(NULL, s.c_string());
异常检查： ASSERT_THROW/ASSERT_ANY_THROW/ASSERT_NO_THROW.
举例：
int Foo(int a, int b)
{
if (a == 0 || b == 0)
{
throw “don’t do that”;
}
int c = a % b;
if (c == 0)
return b;
return Foo(b, c);
}

​ TEST(FooTest, HandleZeroInput)
{
EXPECT_ANY_THROW(Foo(10, 0));
EXPECT_THROW(Foo(0, 5), char*);
}
入参检查：
可以用于打印出错的时候，函数的入参是多少。
ASSERT_PRED1(pred1, val1);
ASSERT_PRED2(pred2, val1, val2);
…
一直到ASSERT_PRED5，目前谷歌支持到5个入参
举例：
bool MutuallyPrime(int m, int n)
{
return Foo(m , n) > 1;
}

​ TEST(PredicateAssertionTest, Demo)
{
int m = 5, n = 6;
EXPECT_PRED2(MutuallyPrime, m, n);
}
上面的例子，当MutuallyPrime返回false的时候，会打印出入参m,n的值。
死亡测试：
ASSERT_DEATH(statement, regex);/EXPECT_DEATH(statement, regex);
ASSERT_EXIT(statement, predicate, regex); /EXPECT_EXIT(statement, predicate, regex);
通常在测试过程中，我们需要考虑各种各样的输入，有的输入可能直接导致程序崩溃，这时我们就需要检查程序是否按照预期的方式挂掉，这也就是所谓的“死亡测试”。
举例：
void Foo()
{
int pInt = 0; pInt = 42 ;
}

​ TEST(FooDeathTest, Demo)
{
EXPECT_DEATH(Foo(), “”);
}
追加debug信息：
ASSERT_EQ(x.size(), y.size()) << “Vectors x and y are of unequal length”;
FAIL()/ADD_FAILURE()/SUCCEED() 分别是致命错误，非致命错误，成功。

​ 深度定制之打印格式：
打印信息的格式也可以自定义，以ASSERT_PRED_FORMAT3(pred_format, val1, val2, val3);为例。
pred_format格式是可以自定义的。代码举例：
testing::AssertionResult AssertFoo(const char m_expr, const char n_expr, const char* k_expr, int m, int n, int k) {
if (Foo(m, n) == k)
return testing::AssertionSuccess();
testing::Message msg;
msg << m_expr << “ 和 “ << n_expr << “ 的最大公约数应该是：” << Foo(m, n) << “ 而不是：” << k_expr;
return testing::AssertionFailure(msg);
}

​ TEST(AssertFooTest, HandleFail)
{
EXPECT_PRED_FORMAT3(AssertFoo, 3, 6, 2);
}

失败时，输出信息：

深度定制之全局设定：
有时候为了测试，需要实现全局的运行环境的准备，这时候可以这么做：
写一个类，继承testing::Environment类，实现里面的SetUp和TearDown方法。

        1. SetUp()方法在所有案例执行前执行
        2. TearDown()方法在所有案例执行后执行
        class FooEnvironment : public testing::Environment
        {
        public:
            virtual void SetUp()
            {
                std::cout << "Foo FooEnvironment SetUP" << std::endl;
            }
            virtual void TearDown()
            {
                std::cout << "Foo FooEnvironment TearDown" << std::endl;
            }
        };
        int main(int argc, _TCHAR* argv[])
        {
            testing::AddGlobalTestEnvironment(new FooEnvironment);    //注意初始化的时候要加这句
            testing::InitGoogleTest(&argc, argv);
            return RUN_ALL_TESTS();
        }
    深度定制之TestSuite环境设定（Fixture TEST_F）
        我们需要写一个类，继承testing::Test，然后实现两个静态方法
        1. SetUpTestCase() 方法在第一个TestCase之前执行
        2. TearDownTestCase() 方法在最后一个TestCase之后执行
        这里注意，对于同一个TestSuite，其实可以有很多TestCase。此处是针对TestSuite的环境设定，所以只在第一个和最后一个TestCase会执行环境的设定和销毁。
        举例：
        class FooTest : public testing::Test {
         protected:
            static void SetUpTestCase() {
            shared_resource_ = new ;
            }
            static void TearDownTestCase() {
            delete shared_resource_;
            shared_resource_ = NULL;
            }
            // Some expensive resource shared by all tests.
            static T* shared_resource_;
        };
        TEST_F(FooTest, Test1)//在编写测试案例时，我们需要使用TEST_F这个宏
         {
            // 此处可以引用之前初始化过的资源
        }
        TEST_F(FooTest, Test2)//在编写测试案例时，我们需要使用TEST_F这个宏
         {
            // 此处可以引用之前初始化过的资源
        }
    深度定制之TestCase环境设定（Fixture TEST_F）
        TestCase事件是挂在每个案例执行前后的，实现方式和上面的几乎一样，不过需要实现的是SetUp方法和TearDown方法：
        1. SetUp()方法在每个TestCase之前执行
        2. TearDown()方法在每个TestCase之后执行
        举例：
        template <typename E> // E is the element type.
        class Queue {
         public:
            Queue();
            void Enqueue(const E& element);
            E* Dequeue(); // Returns NULL if the queue is empty.
            size_t size() const;
            ...
        };
        此时自定义TestCase的环境：
        class QueueTest : public ::testing::Test {
         protected:
            virtual void SetUp() {
            q1_.Enqueue(1);
            q2_.Enqueue(2);
            q2_.Enqueue(3);
            }
            // virtual void TearDown() {}
            Queue<int> q0_;
            Queue<int> q1_;
            Queue<int> q2_;
        };
        运行测试代码：
        TEST_F(QueueTest, IsEmptyInitially) {
            EXPECT_EQ(0, q0_.size());
        }

        TEST_F(QueueTest, DequeueWorks) {
          int* n = q0_.Dequeue();
          EXPECT_EQ(NULL, n);

          n = q1_.Dequeue();
          ASSERT_TRUE(n != NULL);
          EXPECT_EQ(1, *n);
          EXPECT_EQ(0, q1_.size());
          delete n;

          n = q2_.Dequeue();
          ASSERT_TRUE(n != NULL);
          EXPECT_EQ(2, *n);
          EXPECT_EQ(1, q2_.size());
          delete n;
        }
        以上代码，QueueTest对象被建立和摧毁两次，因为有两个QueueTest的TestCase。
    深度定制之解决测试case入参值繁多的问题（参数化TEST_P）：
        有时候测试的入参，需要尝试非常多的值和组合
        TEST(IsPrimeTest, HandleTrueReturn)
        {
            EXPECT_TRUE(IsPrime(3));
            EXPECT_TRUE(IsPrime(5));
            EXPECT_TRUE(IsPrime(11));
            EXPECT_TRUE(IsPrime(23));
            EXPECT_TRUE(IsPrime(17));
        }
        假如需要测试的入参有50个，100个，怎么办？这时候真正进入googletest比较强的环节，根据自定义的规则自动填入参！
        1.你必须添加一个类，继承testing::TestWithParam<T>，其中T就是你需要参数化的参数类型，比如上面的例子，我需要参数化一个int型的参数
        class IsPrimeParamTest : public::testing::TestWithParam<int>
        {
        };
        2.使用INSTANTIATE_TEST_CASE_P这宏来告诉gtest你要测试的参数范围：
        INSTANTIATE_TEST_CASE_P(InstantiationName, IsPrimeParamTest, testing::Values(3, 5, 11, 23, 17));
        第一个参数是测试案例的前缀，可以任意取。 
        第二个参数是测试案例的名称，需要和之前定义的参数化的类的名称相同，如：IsPrimeParamTest 
        第三个参数是重点，作用是参数生成器，上面的例子使用testing::Values表示使用括号内的参数。
        除了testing::Values，Google提供了一系列的参数生成的函数：
        Range(begin, end[, step])    范围在begin~end之间，步长为step，不包括end
        Values(v1, v2, ..., vN)    v1,v2到vN的值----------就是上面例子中用的接口
        ValuesIn(container) and ValuesIn(begin, end)    从一个C类型的数组或是STL容器，或是迭代器中取值。
            举例：
            const char* pets[] = {"cat", "dog"};
            INSTANTIATE_TEST_CASE_P(AnotherInstantiationName, FooTest,
                                    ::testing::ValuesIn(pets));
            结果就是从猫狗字符数组中依次取出猫和狗作为入参。
        Bool()    取false 和 true 两个值
        Combine(g1, g2, ..., gN)    这个厉害，它将g1,g2,...gN进行排列组合，g1,g2,...gN本身是一个参数生成器，每次分别从g1,g2,..gN中各取出一个值，
                    组合成一个元组(Tuple)作为一个参数说明：这个功能只在提供了<tr1/tuple>头的系统中有效。gtest会自动去判断是否支持tr/tuple，
                    如果你的系统确实支持，而gtest判断错误的话，你可以重新定义宏GTEST_HAS_TR1_TUPLE=1。
        官方例子中的Sample #7和Sample #8比较详细的举了相关的例子。
        3.最后，实现TEST_P
        TEST_P(IsPrimeParamTest, HandleTrueReturn)
        {
            int n =  GetParam();        //只要用GetParam()接口，即可获得参数
            EXPECT_TRUE(IsPrime(n));
        }
