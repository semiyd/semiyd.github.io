VNDK overview

VNDK主要解决的是vendor和system分区之间的兼容性问题。
8.0以后，system和vendor是隔离的。
理论上system的进程不会调用vendor的动态库。
vendor的一般也不会调用system的动态库。
但在少数情况下，vendor仍然需要依赖system的动态库。
但treble以后，system是可以独立升级的。
但vendor保持不变。
那如果system内部的动态库换了。vendor的进程运行的时候就可能有问题。

处理这种依赖性问题，有两个办法：
1.保持system内部这些，被vendor依赖的动态库，API不变。
这样做成本很高。
但好处是可以避免system和vendor进程载入不同版本的动态库。
动态库在内存里面只有一份。
2.在升级system的时候，把老的system动态库多拷贝一份，放在新的升级的system镜像中。
但这样做有个前提。就是system中调用新库的进程，和vendor中调用老库的进程。互相之间的通信，要非常小心。
要保证使用一个稳定的接口。例如HIDL。
另外从新库和老库之间传递的对象，可能实际定义已经发生了改变，导致对象的解析出问题。

所以针对上面这些不同的场景。system下的共享库，分成几类：
LL-NDK
就是那些保证API不变的库。主要是opengl和c库。
libEGL.so, libGLESv1_CM.so, libGLESv2.so, libGLESv3.so, libandroid_net.so, libc.so, libdl.so, liblog.so, libm.so, libnativewindow.so,
libneuralnetworks.so, libsync.so, libvndksupport.so, and libvulkan.so
VNDK
就是上面说的，可以多拷贝一份老库的库。
正因为system镜像链接新库，vendor镜像链接老库，所以这种拷贝有如下的前提：
It does not send/receive IPCs to/from the framework.
It is not related to ART virtual machine.
It does not read/write files/partitions with unstable file formats.
It does not have special software license which requires legal reviews.
Its code owner does not have objections to vendor usages.
安装位置是：
/system/lib[64]/libexample.so给system用的
/system/lib[64]/vndk/libexample.so给vendor用的
FWK-ONLY
这类就是system内部专用的framework的库。vendor进程是严禁使用的。

VNDK是有版本号的。其ro.vndk.version存放于/vendor/default.prop
