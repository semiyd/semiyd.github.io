拷贝长度为n的字节：

static inline unsigned long __must_check copy_to_user(void __user to, const void from, unsigned long n)
以上两个API，均返回不能复制的字节数。如果完全成功，返回0.如果复制失败，返回负值。

copy_from_user和copy_to_user内在实现机制的详细解析：
函数在arch/arm/include/asm/uaccess.h

 static inline unsigned long __must_check copy_from_user(void *to, const void __user *from, unsigned long n)
 {
     if (access_ok(VERIFY_READ, from, n))
         n = __copy_from_user(to, from, n);
     else /* security hole - plug it */
         memset(to, 0, n);
     return n;
 }
 static inline unsigned long __must_check copy_to_user(void __user *to, const void *from, unsigned long n)
{
     if (access_ok(VERIFY_WRITE, to, n))
         n = __copy_to_user(to, from, n);
     return n;
 }

从代码上看。就分成两步。
第一步是用access_ok检查地址合法性。
第二步是用__copy_from_user/__copy_to_user去真正拷贝数据。
下面来分析这两部。
1.access_ok()
    这个的定义位于arch/arm/include/asm/uaccess.h
    #define access_ok(type,addr,size)    (__range_ok(addr,size) == 0)    
    #define __range_ok(addr,size) ({ \
        unsigned long flag, roksum; \            
        __chk_user_ptr(addr);    \
        __asm__("adds %1, %2, %3; sbcccs %1, %1, %0; movcc %0, #0" \
            : "=&r" (flag), "=&r" (roksum) \
            : "r" (addr), "Ir" (size), "0" (current_thread_info()->addr_limit) \
            : "cc"); \
        flag; })
    首先access_ok()实际就是__range_ok()
    flag保存结果的变量：非零代表地址无效，零代表地址可以访问。roksum变量：保存要访问的地址范围末端，用于和当前进程地址空间限制数据做比较。
    __chk_user_ptr(addr);这个的定义位于include\linux\compiler.h
        那么从compiler.h可以看到。如果定义了宏__CHECKER__，则__chk_user_ptr声明为extern void __chk_user_ptr(const volatile void __user *);
        如果没定义宏__CHECKER__，则__chk_user_ptr声明为(void)0也就是空函数。
        那么这个宏__CHECKER__，其实只有在用sparse工具检查代码的时候，才会打开，默认都是关闭的。
        所以平常情况下，其实是空函数。
        当宏__CHECKER__打开的时候，声明为extern void __chk_user_ptr(const volatile void __user *);，依然是空函数，只是带上了参数。
            而这个__user，sparse会根据这个关键字，检查指针是否在用户地址空间。（sparse具体如何做到检查__user？暂不清楚）
        有关Sparse，再多提几句：
            sparse是kernel代码的静态检查工具，内核make的时候会被运行的。
            sparse支持很多关键字修饰。比如__user,__kernel,__iomem,这些关键词都是给sparse用的，其实对于gcc毫无意义。
            sparse会根据关键字，决定检查指针是不是用户空间的，是否是内核空间的，等等，或者是否存在把用户空间的指针，赋值给声明为内核空间的指针。
        总之。这个函数会导致sparse时候检查addr是否是用户空间指针。但真正运行时候，这个函数实际为空。
    然后分析下面这段汇编：
        __asm__("adds %1, %2, %3; sbcccs %1, %1, %0; movcc %0, #0" \
            : "=&r" (flag), "=&r" (roksum) \
            : "r" (addr), "Ir" (size), "0" (current_thread_info()->addr_limit) \
            : "cc"); \    
        具体内联汇编的含义暂不分析了。可以参考Linux\Kernel\学习资料\基于ARM 构架（带MMU）的copy_from_user与copy_to_user详细分析.pdf
    总之就是意思是：
        如果（addr + size）>=（current_thread_info()->addr_limit），返回非零值
        如果（addr + size）<（current_thread_info()->addr_limit），返回零
    也就是说，检查保存要访问的地址范围末端，是否超过当前进程地址空间限制。

2.__copy_from_user/__copy_to_user
    这个的定义位于include/asm-generic/uaccess.h，对于ARM构架，没有单独实现这两个函数。
    以__copy_from_user为例：
    static inline __must_check long __copy_from_user(void *to,
            const void __user * from, unsigned long n)
    {
        if (__builtin_constant_p(n)) {        //GCC的内建函数,用于判断一个值是否为编译时常数，如果参数值是常数，函数返回 1，否则返回 0。
            switch(n) {                //对于拷贝字节数是1，2，4，8的，直接用地址对地址拷贝。
            case 1:
                *(u8 *)to = *(u8 __force *)from;
                return 0;
            case 2:
                *(u16 *)to = *(u16 __force *)from;
                return 0;
            case 4:
                *(u32 *)to = *(u32 __force *)from;
                return 0;
    #ifdef CONFIG_64BIT
            case 8:
                *(u64 *)to = *(u64 __force *)from;
                return 0;
    #endif
            default:
                break;
            }
        }

        memcpy(to, (const void __force *)from, n);    //否则如果不是常数或n过大，则使用memcpy函数
        return 0;
    }
所以基本上看出来，其实copy_from_user和copy_to_user真正区别于memcpy的特点在于增加了access_ok()检查。
如果没有access_ok()，用户空间的程序可以任意指定一个内核空间的地址，来完成读写，造成严重后果。

问题：为什么不能直接用memcpy？而要用copy to/from user这种API呢？
1.mcu模式
在ARM上就是kernel运行在SVC模式(最高级)，App运行在user模式。当user app执行一个system call调用kernel代码，kernel会完成CPU的模式切换。
SVC模式能访问的寄存器更多，如果在kernel中对user app传入的地址直接访问，可能会有以下问题：传入的地址是错误的，则kernel也可以访问，可能会破坏其他kernel内容，这会引起安全问题
2.地址范围
如果没有access_ok()，用户空间的程序可以任意指定一个内核空间的地址，来完成读写，造成严重后果。
